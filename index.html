需求供给理论——均衡价格理论前奏

Posted on 2018-05-26 | In Course

需求理论

定义

在定义中需要明确需求是一种关系，是在以特定时期内，一定价格与在此一定价格之下消费者愿意且能够购买的商品数量之间的关系；而需求量是一种具体的值，是消费者愿意且能够购买的商品的数量。

需求规律

  • 价格和需求量成反比。
  • 但是奢侈品和吉芬商品不符合此规律。

    替代品：一种商品的价格与另一种商品的需求正向变动；
    互补品：一种商品的价格和另一种商品的需求反向变动。

需求函数

很多因素会影响需求数量，比如P(价格)Pr（相关商品的价格）、M（收入水平）、F（对未来收入的预期）……为分析简单，就只分析价格（P）一种因素
$Q=f(P,Pr,M,F……)$
Q=f(P)

如果价格不变，只考虑相关商品对需求量的影响，有需求交叉函数：
Q=f(Pr)

类似还有需求收入函数

需求与需求量的变动

  • 需求的变动是需求曲线的移动，而需求量的变动是需求曲线上的点在曲线上的移动。

  • 需求变动是受到价格之外的因素影响，需求量的变动是只受到价格影响引起的需求量的变动。

需求弹性

有点弹性和弧弹性两种，点弹性用到微分的形式，但其实两个式子的形式是差不多的。

  • 需求价格弹性

是需求对于价格变化的反应程度，弹性系数等于需求量变动的百分比除以价格变动的百分比。
对于线性需求而言，斜率不变，但是弹性是一直变动的，因为P/Q在不同的位置值不一样。

  • 分类

富有弹性ed>1 需求量对价格的反应比较敏感,降价可以增加销售收入

缺乏弹性ed<1 降价会减少收入

单位弹性ed=1， 这时价格变动1%引起的需求量变动也是1%，价格变化对于销售收入没有影响

没有弹性ed=0，其需求线与纵轴平行

完全弹性ed趋于无穷，其需求线与横轴平行

吉芬商品和炫耀性商品的需求价格弹性大于0

  • 需求交叉价格弹性

0替代品

<0互补品
=0无关商品

  • 需求收入弹性

与需求价格弹性定义、算法类似

eM>1,奢侈品,需求对于收入变化较为敏感
eM<1,必需品
eM<0,抵档商品，当收入增加时会减少对此类商品的需求，但是到后面会知道在替代效应和收入效应共同影响之下，抵档商品的总效应为正。

供给理论

  • 供给价格弹性

eS>1,富有弹性

<1,缺乏弹性
=1，单位弹性
=0，没有弹性
=a，完全弹性

（与需求价格弹性基本一样）

微观经济学(一)

Posted on 2018-05-24 | In Course
  • 为什么边际成本线高于平均总成本曲线时，平均总成本曲线呈上升趋势？

MC是一条神奇的曲线，它交与ATC和AVC的最低点。边际成本大于平均总成本时，平均总成本就会上升，或者换一种说法：当边际成本经过平均总成本曲线后，边际成本的上升带动了平均总成本的上升。（随着边际产量先递增后递减的规律，边际成本和平均可变成本也先递减后递增）想一下，当平均总成本为一个值时，边际成本是多生产一个产品所产生的成本，当边际成本增加时，平均总成本也跟着增加。

  • 在完全竞争市场下，价格由市场决定企业无法决定，因此价格表现在图上就是一条不变的水平直线，但是企业终归是要寻求利益最大化即利润最大化。在$P=MR=MC$时利润实现最大化，推导过程如下。

企业的利润计算公式 $L=P*Q-C(Q)$

要求最大利润，对R求导 $L’=P-dC/dQ$

令$R’=0$ 有 $P=dC/dQ=MC$

这表明P=MC时利润最大，此时的产量可使利润最大化，而在完全竞争市场下价格不变，因此边际收益MR=P。

所以 $P=MR=MC$

Java基础（一）

Posted on 2018-05-14 | In Java

  • 面向对象的理解并举例？
    面向对象是基于面向过程而言的，可以把复杂的事情简单化，让我们从执行者变为指挥者每提高了代码的复用性。
    例如：烧水（面向过程：找到烧水壶——打开盖子——装水——关闭盖子——打开电源——水烧好了） （面向对象：拿壶——装水——烧水——等待——水烧好了）

  • 类与对象之间的关系？
    类就是有很多相同特性的分类，而对象就是一类中具体的一个。

  • 如何对类进行分析，如果创建自定义类对象，并如何指挥对象做事情？
    首先分析这个类是个什么样的事物；其次分析这个类具有哪些属性和行为；把这个事物的属性和行为对应成我们的成员变量和成员方法。
    通过new关键字创建这个类的对象。 类名.属性名; 类名.方法名（）；直接调用。

  • 成员变量和局部变量的区别？
    作用范围
    成员变量：作用于整个类中。 局部变量：作用于所在区域。
    生命周期
    成员变量：随着类的创建而存在，随着类的消失而消失，执行完语句后等JVM空闲时由垃圾回收机制清理。
    局部变量：是创建对象时才存在的，执行完回立即释放；所以成员变量的生命周期更长。
    初始值
    成员变量：存在于堆内存中，有默认初始值。 局部变量：存在于栈内存中，没有默认初始值，需要手动赋值。

Java中函数的定义

Posted on 2018-05-07 | In Java
  • 函数的格式

修饰符 返回值 类型函数名(形式参数类型1 参数名1,形式参数类型2 参数名2,形式参数类型3 参数名3……)

{

函数体语句;

return 返回值;

}

  • 修饰符：public 公共的static 静态的

  • 返回值类型：就是函数所实现的功能结果的数据类型。

  • 方法名：自己定义，符合标识符命名规则就行；驼峰命名规则 :单个单词首字母小写，多个单词后面的单词首字母大写。

  • 参数列表：即形式参数列表，用于接收实际参数。

  • 参数类型：参数的数据类型

  • 参数名：即变量名

  • 函数体语句：就是函数要完成的功能

  • return:用来结束函数

  • 返回值：即这个函数的功能的结果 ，由return带回调用地方。

    注意:void 表示不确定返回值类型，不能理解为没有返回值，只是返回值的类型不确定而已;

    此外，返回值类型为void时，由于其返回值类型不能确定，故不能定义形式参数变量来接收；

    当返回值类型为void时，return可以不写；但是当返回值类型为其他如int String等时就必须写，否则编译会出错。

  • 自定义一个函数的三步

    • 确定返回值类型

    • 确定方法名

    • 明确参数列表

  • 函数的参数传递

    • 形式参数：函数定义时的参数为形式参数——>简称形参

    • 实际参数：函数调用时的参数为实际参数 ——>简称实参

      参数传递：只能有实参传给形参

      注意：实参和形参在个数、顺序、类型上都要保持一致

AnLine——T系列区块链简述

Posted on 2018-04-23 | In Insurance

互联网保险公司众安保险已经对区块链技术进行了某些方面的应用，企业团队可以基于众安这个区块链云平台Anlink构建相关领域的应用，免去搭建底层技术的精力和成本。

而在安链云平台上存在着T系列区块链，T系列区块链主要包括钛合约、钛存证、钛空舱和钛阳四部分。它们分别对应的是电子签章、安全性、存储、唯一性(授权第三方)。

  • 钛合约

钛合约可以说是电子签章，是签名阶段。它有四个特点：一是随时随地能够让客户进行签名，利用互联网为客户创造了便利，同时也为企业节省了人力成本。二是保证信息的真实性，通过区块链上的信息共享打通公安、银联、企业工商信息，从而保证客户填写的信息的真实性。三是对客户隐私的尊重，对用户所填文件加密，填写过程不接触用户原文。四是利用区块链的不可篡改性进行区块链存证，保护信息的安全性。

  • 钛存证

钛存证是对区块链的典型应用，哈希指纹、去中心化、信息溯源。这些都是对信息安全的保护。

  • 钛空舱

主要是存储方面的功能，对文件进行分布式存储，，并多节点备份还原 ，适应灾备需求。信息访问安全，访问存储资源需通过服务网关授权访问，保护信息的安全性。

  • 钛阳

强调的是文件信息的唯一性，链上账户统一管理，用户将属性鉴权授予第三方，之后在用户授权下，不同机构可以共享用户的鉴权信息。唯一的信息在链上却可以供很多人用，但是有一个问题还不太清楚，就是多方机构获取属性鉴权，那么用户的信息是否随时都能被查看呢？

区块链部分弊端——骗保的应用

Posted on 2018-04-22 | In Insurance

区块链最近大热，它具有去中心化，不可篡改性和数据可追溯，分布式记账等的特点，然而，它同时也具有一些缺点是无法回避的。下面主要说的是在骗保方面区块链表现出来的几方面的弊端。

  • 信任源的问题

区块链上的信息具有不可篡改性，如果要篡改必须要有51%的攻击才行。这主要运用到哈希和块。具体可详见：

https://cloud.tencent.com/developer/article/1058084

虽然不可篡改听起来是对于信息的一种安全保护，但是，如何保证客户在一开始输入的信息的正确性？如果以后需要修改信息怎么办？只有原始信息正确，智能合约才能正确执行。因此信息源的问题还在于如何完成科技与人的完美对接。

  • 隐私问题

客户的私密信息只有客户的密钥才能查看，那么如果有人获得你的密钥，那你的信息岂不是就会暴露，完全没有隐私可言，或者是本人忘记了密钥，如何补救？在这种担忧下，客户是否会输入真实可靠的信息又受到了质疑。以及密钥是否会一直完全不变，客户本人是否能够变更？

  • 对于去中心化是不是真的好

去中心化在我理解相当于一个国家的无政府状态，这种状态一般来说是不会很好的，当然也又好的例子。去中心化的话，如果出现触犯法律的情况，该如何制裁。
区块链上的块具有相互独立性，例如比特币所有节点都遵循着一个共识算法，比特币系统则是每一个“块”都是由所有节点通过算力竞争产生的。依靠共识算法来选择数据信任源是一种去中心化的方法，但是当一个人掌握的计算能力超过整个比特币网络的51%，他就成为了事实上的比特币的中心。而面对此种情况相关法律法规还不完善，很难得到解决。

  • 它的延迟性可能会因为数据越来越多，数据链越来越长而越来越明显。尽管这貌似在保险的骗保业务中影响不是很大。

  • 对于区块链的应用所带来的收益是否值得企业家所投入的成本，这直接决定了区块链的发展和应用。同时，社会和民众是否能够接受这一技术的变革还有这一领域技术上的难题都是区块链应用上的绊脚石。只要第三个问题解决了，前两个问题自然也就引刃而解了，不过，技术上的不成熟是需要不断探索的，所以各公司花费大量的人力物力财力去攻克技术上的难关是不太可能的。

  • 区块链还会受到电力，网络的制约。

车险骗保

Posted on 2018-04-19 | In Insurance

目前，保险行业的骗保现象屡见不鲜，车险骗保率约24%-40%，车险骗保率高的原因一般来说是骗保利润高，风险小，难度低等原因。

那么骗保手段则花样不断，种类繁多，不过，还是离不开车主，维修店，保险公司人员，相关律师，保险公司以及相关协会、部门政府这几方的参与。骗保的手段从参与者方面来说是这些参与方的排列组合所形成的。

  • 车主本人有意骗保，自行行动进行骗保。

此种案例大多是车主本人事故后伪造成骗保所需场景，也有一种情况则是先出险后保险。这两种情况的参与方主要有车主本人，交通部门，保险公司定损员，保险公司以及相关部门。这两种情况在骗保行为当中是比较容易识别真伪的，如果这类骗保行为成功则可能有定损员的过失，或者是相关部门对调查期限的限制导致保险公司草草了事。
若采用区块链技术，利用其信息的不可篡改性，对车主的信用进行记录，则可以有效防止车主连续骗保行为的发生。对于先出险后保险的行为也可以有效防止。

  • 车主和维修公司组合骗保。

这种组合情况相对来说是较为普遍的，同时也是较为成功的，甚至一些维修公司的很大一部分收入来源是利用其代理车险的服务拿车主的车进行骗保赚取中间差价，显然其中利润可观。
车主和维修公司联合又分两种情况：一种是车主即当事人不知情，另一种是当事人之情或默许。

前者中维修公司会给车伪造出更严重的事故，或以次充好，然后到保险公司办理保险服务，在这一过程中车主完全不知情。或者是车主因办理保险赔偿较为繁琐，直接委托维修公司去保险公司，这都是造成骗保行为的原因。在这种情况当中，保险公司若能降低办理成本，简化程序，让当事人更加自愿亲自办理，骗保行为也能减少。而区块链的分布式记账，公开，不可篡改的特点都有利于保险公司解决取证难的问题。在查证过程中，可以直接从区块链上调取相关信息，不用等待相关部门的回复审批等，从而也会节省时间。同时也不用担心信息的虚假性，因为区块链上的信息一旦被篡改会被发现，并且篡改信息是很困难的，成本巨大，骗保人也绝对不会因为几千或几万的保费而花费如此巨大的成本。车主也可以直接在区块链上填写相关信息，减少车主办理保险赔偿的成本。车主的信息是要本人提供密钥才可以查看的，所以当事人不用因为担心信息泄露而少填或不填，从而导致当事人与保险公司之间信息不对称，降低事件解决效率。

后者车主和维修公司联合骗保，很多车主都是由于贪小便宜，不懂相关法律知识，先被维修公司或所谓的中间人欺骗可以免费维修车辆，之后为维修公司提供一系列证件等，就演变成了和维修公司一起犯案。对此，保险公司如果可以对此类维修公司进行调查，了解其信用历史，也可以了解车主的所要保费的历史记录，对两者做出一些是否骗保的判断，会对辨别是否骗保具有重要意义。因为大多数的骗保行为达不到法律规定的限度，因此相关执法部门并不能给予保险公司相关帮助，从而保险公司造成损失。若保险公司能够自己解决一些取证需要，骗保行为就能够更快被识别出来。

  • 车主、维修公司和保险公司人员的“一条龙”服务。

今天有“独狼式”的骗保，也有“一条龙”产业链式的骗保，有专门的“撞车手”，也有专门主事的汽修店老板，也有专门从事保险理赔的无良律师，还有里应外合的定损员。这些人形成了一个骗保链条，有些汽修公司还雇佣一批定损员等和理赔相关的人员，在定损时蒙混过关拿到保费。
